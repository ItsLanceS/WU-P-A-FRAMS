SYSTEM ALGORITHM ANALYSIS
Project: FRAMS (Face Recognition Attendance Management System)
Date: April 23, 2026

1. Introduction
FRAMS is a hybrid PHP + Python offline face-recognition attendance system. It solves the problem of fast, low-contact, and auditable attendance logging for students by combining webcam capture, local face matching, and database-backed attendance records.

The algorithm analysis is important because recognition quality directly affects attendance correctness (false accepts and false rejects), while computational cost affects real-time user experience in classroom conditions.


2. System Overview
2.1 Architecture
- Frontend (JavaScript): Captures webcam frames and sends recognition requests.
- PHP MVC backend: Handles application logic, student/event validation, and attendance writes.
- Python offline API (Flask + OpenCV + NumPy + PIL): Performs enrollment and face matching.
- Storage:
  - MySQL: students, attendance, events.
  - JSON file (python_api_offline/data/enrollments.json): enrolled face templates.

2.2 Inputs
- Enrollment input: student_id, name, and multiple base64 image samples.
- Recognition input: one base64 webcam frame, action (time_in/time_out), optional event_id.
- Attendance constraints: active session, valid student, confidence >= system threshold, valid active event for date.

2.3 Outputs
- Recognition result: matched student_id (or null), confidence, matcher type, distance and gap, face metadata.
- Attendance result: time_in/time_out status, duplicate warnings, invalid-event warnings.

2.4 Workflow Placement
- Enrollment workflow:
  1) Student images captured/uploaded in PHP UI.
  2) PHP sends images to Python /enroll.
  3) Python extracts templates and stores them in enrollments.json.
- Recognition workflow:
  1) Webcam frame sent from JS -> PHP API endpoint.
  2) PHP forwards frame to Python /recognize.
  3) Python returns match + confidence.
  4) PHP validates threshold/event/record state.
  5) PHP writes time_in or time_out in MySQL.


3. Algorithm Description
3.1 Enrollment Algorithm (Python /enroll)
For each input image:
1) Decode base64 and load RGB image.
2) Detect faces using Haar Cascade.
3) Select largest detected face.
4) Build two hash samples:
   - Face-cropped perceptual hash.
   - Full-frame perceptual hash (backward-compatible fallback).
5) Build two descriptor samples:
   - Face-cropped descriptor.
   - Full-frame descriptor.
6) Aggregate all valid samples.
7) Upsert student profile in enrollments.json.

3.2 Recognition Algorithm (Python /recognize)
1) Decode and load frame.
2) Detect faces; choose largest face.
3) Build incoming candidates:
   - 1-2 hashes (face and optional full frame).
   - 1-2 descriptors (face and optional full frame).
4) For each enrolled student:
   - Compute pairwise descriptor distances against incoming descriptor set.
   - Compute pairwise hash Hamming distances against incoming hash set.
   - Reduce each metric using representative_distance (mean of top-k best distances, k=3).
5) Sort students by descriptor distance and hash distance.
6) Apply acceptance checks:
   - Descriptor path first: distance, confidence, and winner-vs-runner-up gap thresholds.
   - If descriptor fails, try hash path with corresponding thresholds.
7) Return best confident match or "No confident match".

3.3 Attendance Decision Algorithm (PHP)
1) Receive recognition result from Python.
2) Validate session/auth and confidence threshold.
3) Resolve student_id to active student record.
4) Validate optional event_id and event date/activity.
5) Read existing attendance for (student, date, event).
6) For action=time_in:
   - If no record, insert time_in.
   - Else return already_timed_in warning.
7) For action=time_out:
   - If no time_in record, return missing_time_in warning.
   - If time_out exists, return already_complete warning.
   - Else update time_out.

3.4 Key Assumptions
- Largest detected face is the primary subject.
- Multiple enrollment images improve template robustness.
- Descriptor metric is more discriminative than hash metric, so it is evaluated first.


4. Data Structures Used
- Arrays/Lists:
  - Python lists for faces, hashes, descriptors, and per-student distances.
  - Chosen for fast append, iteration, and sorting in batch comparisons.
- Dictionaries/Associative objects:
  - Student records and API payloads as key-value maps.
  - Chosen for clarity and direct field access (student_id, confidence, distance, etc.).
- JSON document store:
  - enrollments.json containing students[] with hashes/descriptors.
  - Chosen for offline simplicity and easy serialization.
- Relational tables (MySQL):
  - attendance and student/event entities.
  - Chosen for transactional consistency, filtering, and reporting.
- Strings as bit-vectors:
  - 64-bit image hash represented as string of '0'/'1'.
  - Supports simple Hamming distance computation.
- Numeric vectors:
  - Face descriptors are normalized float vectors.
  - Support cosine-based distance matching.


5. Algorithm Complexity Analysis
Let:
- S = number of enrolled students.
- H = average hashes per student.
- D = average descriptors per student.
- C_h = number of incoming hashes (<= 2).
- C_d = number of incoming descriptors (<= 2).
- P = number of pixels processed in detection/feature extraction pipeline.
- K = top-k reduction constant (3).

5.1 Enrollment Complexity (per request with M input images)
- Face detection + feature extraction per image: O(P).
- Total template generation: O(M * P).
- Upsert search in students list: O(S).
- Overall time: O(M * P + S).
- Space: O(M * (hash_count + descriptor_length)) for temporary samples; plus persistent profile size.

5.2 Recognition Complexity (per request)
- Preprocessing + face detection + incoming features: O(P).
- Descriptor comparisons across all students:
  O(S * D * C_d * L_d), where L_d is descriptor length (constant for this system).
- Hash comparisons across all students:
  O(S * H * C_h * L_h), where L_h = 64 bits (constant).
- Sorting candidates:
  O(S log S) for descriptor and O(S log S) for hash.
- Overall dominant time:
  O(P + S * (D + H) + S log S) with descriptor/hash lengths treated as constants.

5.3 Best/Average/Worst Case (Recognition)
- Best case:
  - Small S, clear winner, one face detected.
  - O(P + S) comparison dominant with low constants.
- Average case:
  - Typical class roster with both descriptor and hash candidate evaluation.
  - O(P + S * (D + H) + S log S).
- Worst case:
  - Large S, many templates per student, low separability requiring full threshold evaluation.
  - O(P + S * (D + H) + S log S) with larger constants and memory pressure.

5.4 Attendance Write Complexity
- Lookup + conditional insert/update in DB is near O(1) average with proper indexing on (student_id, date, event_id).
- End-to-end write path effectively O(1) relative to recognition cost.


6. Performance Analysis
6.1 Efficiency
- Real-time behavior is achieved by periodic frame scanning (configurable interval in UI) instead of every video frame.
- Main CPU costs are face detection and descriptor extraction; matching cost scales linearly with enrolled templates.

6.2 Scalability
- Current scaling characteristic is approximately linear with total stored templates.
- JSON-file template storage is simple but can become a bottleneck as S grows (read/parse/write overhead and no indexed nearest-neighbor search).

6.3 Resource Utilization
- CPU: moderate to high during scan windows (OpenCV + NumPy operations).
- Memory: proportional to template count loaded from JSON plus temporary arrays.
- I/O: enrollment updates rewrite JSON store; recognition reads store each request.

6.4 Practical Observations for This Design
- Descriptor-first then hash-fallback improves robustness over hash-only matching.
- Gap-based thresholding reduces ambiguous matches in close-score scenarios.
- Event-aware attendance checks prevent incorrect marks outside valid schedule.


7. Comparative Analysis
Approach A: Current Hybrid (Descriptor-first + Hash fallback)
- Advantages:
  - Better recognition robustness than hash-only.
  - Backward compatibility via legacy hash path.
  - Transparent confidence and distance outputs for debugging.
- Disadvantages:
  - Linear scan over all templates.
  - Heuristic thresholds require tuning.

Approach B: Hash-only Matching
- Advantages:
  - Simpler implementation, lower compute.
- Disadvantages:
  - Sensitive to illumination/pose; weaker discriminative power.

Approach C: Deep Embedding + ANN Index (e.g., FAISS)
- Advantages:
  - Better accuracy and faster retrieval at larger scale.
- Disadvantages:
  - More dependencies, operational complexity, and model management overhead.

Approach D: Cloud Recognition Service
- Advantages:
  - Potentially higher accuracy and managed scalability.
- Disadvantages:
  - Requires network connectivity, recurring cost, and data-governance considerations.

Trade-off Summary:
Current FRAMS algorithm is a practical offline compromise: better than naive hash methods, lighter than full deep-learning infrastructure, and suitable for small-to-medium deployments.


8. Limitations and Assumptions
- Largest-face assumption may fail in crowded scenes.
- Haar cascades can degrade in difficult lighting/angles compared with modern deep detectors.
- JSON store is not ideal for high concurrency or very large enrollment sets.
- Thresholds are environment-specific and may require periodic recalibration.
- No explicit liveness/anti-spoofing step is present in the current pipeline.
- Confidence thresholding can reject valid users when camera quality is low.


9. Conclusion and Recommendations
9.1 Conclusion
FRAMS uses a layered offline recognition strategy (descriptor-first, hash-fallback) integrated with strict PHP-side attendance state checks. The algorithm is computationally manageable and functionally suitable for real-time classroom attendance, especially in environments where offline operation is required.

9.2 Recommendations
1) Add liveness detection (blink/motion/challenge-response) to reduce spoof risk.
2) Replace JSON matching store with indexed vector storage or ANN search for larger enrollments.
3) Cache enrollment templates in memory with change tracking to reduce per-request file I/O.
4) Add benchmark suite (latency, FAR/FRR, throughput) across lighting and pose conditions.
5) Introduce adaptive threshold calibration per camera/environment.
6) Consider stronger face detector (DNN-based) for better detection reliability.
